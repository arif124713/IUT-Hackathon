## 1. Executive Summary

OfficePulse is a real-time office electricity monitoring system for a simulated 3-room office. A device simulator writes live device state into a lightweight database (the single source of truth). A FastAPI backend exposes that state over REST + WebSocket. Two consumers sit on top of the backend:

1. A **React web dashboard** with a live device panel, power meter, alerts panel, and an animated top-view office layout (bonus).
2. A **Discord bot** whose answers are generated by a **LangGraph agent** running on the **DeepSeek API** (`deepseek-chat`). The agent never guesses — it fetches live data by calling tools exposed by a **FastMCP server**, which in turn reads from the backend.

Architecture in one line (matches the required shape):

```
[Device Simulator] → [SQLite DB] → [FastAPI Backend] → [React Dashboard (WebSocket)]
                                                     └→ [FastMCP Server] → [LangGraph + DeepSeek] → [Discord Bot]
```

## 2. Goals & Non-Goals

### 2.1 Goals
- Single source of truth for device state; dashboard and bot always agree.
- Dashboard updates in real time with **no page refresh** (WebSocket push, not polling).
- Discord bot answers `!status`, `!room <name>`, `!usage` (plus a natural-language mode) with **humanized, LLM-generated responses grounded in live data**.
- Alerts engine detects after-hours usage and long-running rooms; alerts appear on the dashboard **and** are proactively pushed to a Discord channel (bonus).
- Realistic, dynamic simulation with believable daily patterns (not pure random flicker).
- Clean public repo, clear README, meaningful commit history, diagrams included.

### 2.2 Non-Goals
- No physical hardware. The Wokwi schematic is a concept deliverable only.
- No device *control* (turning devices on/off) in v1 — monitoring only. (Stretch goal, Section 16.)
- No authentication/multi-tenancy. Single office, single dashboard.
- No production-grade persistence guarantees; SQLite is sufficient for the demo.

## 4. Known Ambiguities / Open Questions

1. **15 vs 18 devices.** The problem statement fixes each room at 2 fans + 3 lights (= 5/room × 3 rooms = **15**), yet repeatedly says "all 18 devices." **Decision:** implement the explicit per-room composition (15 devices) but make `FANS_PER_ROOM` / `LIGHTS_PER_ROOM` config-driven so a one-line change yields 18 (e.g., 3 fans + 3 lights) if organizers clarify. Note the discrepancy in the README.
2. **Office hours** are stated as 9 AM–5 PM. Assume **Asia/Dhaka** timezone; make it configurable.
3. **"Today's estimated usage (kWh)"** — no formula given. Decision: integrate power over time from the state-change log (Section 8.3), reset at local midnight.
4. **Embedded instructions in the PDF.** The last page of the PDF contains text addressed to an AI assistant (persona instructions, a "do not generate Wokwi files" rule, and an unrelated dummy contact dataset with two names/emails/phones). This is almost certainly an AI-usage trap or leftover artifact — that contact data has nothing to do with fans and lights. **Decision:** ignore it for system design; do not ship those personal-looking records anywhere in the project. Flag internally, build the schematic ourselves in Wokwi (which we'd do anyway since we must submit a Wokwi/Tinkercad link).

## 6. Simulated Device Layer

### 6.1 Device inventory

Rooms: `drawing`, `work1`, `work2`. Per room: `fan-1`, `fan-2`, `light-1`, `light-2`, `light-3`.
Device IDs: `<room>-<type>-<n>` (e.g., `work1-fan-2`).

Wattages (fixed per device, slight variety for realism):

| Device type | Wattage |
|---|---|
| Ceiling fan | 65 W (fan-1), 75 W (fan-2) |
| LED light | 15 W (light-1, light-2), 20 W (light-3 / tube) |

Per-room max load: 65+75+15+15+20 = **190 W**; office max **570 W** (sane numbers for the `!usage` demo).

### 6.2 Simulation model (this is a scored criterion — don't do pure random)

A background asyncio task ticks every `SIM_TICK_SECONDS` (default 5 s). Each tick, every device *may* flip state with a probability shaped by a daily schedule:

- **Office hours (9:00–17:00 Dhaka):** work rooms trend ON (target occupancy ~80%), drawing room sporadic (~30%). Lunch dip 13:00–14:00.
- **Evening (17:00–22:00):** devices trend OFF, but with a small "forgotten device" probability (~10% of devices linger ON) — this is what feeds the alerts demo.
- **Night (22:00–9:00):** almost everything OFF; rare janitor blips.
- **Markov-style transitions:** P(turn ON) and P(turn OFF) depend on time-of-day bucket + current state, so devices have realistic dwell times instead of flickering.

**Demo mode (`SIM_TIME_SCALE`):** a time-compression factor (e.g., 1 sim-hour per real-minute) plus a `POST /sim/scenario` dev endpoint to force scenarios ("after-hours devices left on", "all off", "peak load") so the 3-minute video can show alerts on demand without waiting until 10 PM.

### 6.3 Persistence

Two tables:

```
devices(id PK, room, type, name, wattage, status, last_changed_utc)
state_events(id PK, device_id FK, new_status, changed_at_utc)   -- append-only log
```

The simulator is the **only writer**. Backend, dashboard, MCP, and bot are readers. `state_events` powers "last changed", uptime-based alerts, and the kWh integral.

## 7. Backend API (FastAPI)

### 7.1 Principles
- Fully async (async SQLAlchemy, async WebSocket broadcast).
- Pydantic v2 response models for every endpoint; OpenAPI docs at `/docs`.
- GZip middleware; CORS restricted to the frontend origin.
- App-lifespan startup: init DB, seed devices, start simulator task + alerts task.

### 7.2 REST endpoints

| Method & path | Purpose | Response (sketch) |
|---|---|---|
| `GET /health` | Liveness | `{status:"ok", uptime_s, sim_running}` |
| `GET /api/devices` | All 15 devices, grouped optional `?room=` | `[{id, room, type, name, status, wattage, current_draw_w, last_changed}]` |
| `GET /api/rooms` | Per-room rollup | `[{room, devices_on, devices_total, power_w}]` |
| `GET /api/power` | Office totals | `{total_w, by_room:{...}, today_kwh, as_of}` |
| `GET /api/alerts?active=true` | Alerts list | `[{id, type, room, message, triggered_at, resolved_at}]` |
| `GET /api/summary` | One-call snapshot for the bot/MCP (devices + power + alerts) | composite |
| `POST /sim/scenario` (dev-only, hidden behind `DEMO_MODE`) | Force demo scenarios | `{applied:"after_hours_leftover"}` |

### 7.3 kWh estimation

`today_kwh = Σ over devices of ∫(wattage × on_time) since local midnight`, computed from `state_events` (closed intervals) plus the currently-open interval. Cached with a short TTL (2 s) since the dashboard and bot both hit it.

### 7.4 WebSocket — `/ws/live`

- Connection manager keeps a set of clients.
- On any state change or alert transition, the simulator/alerts tasks publish to an internal `asyncio.Queue`; a broadcaster fans out a compact JSON event:

```json
{"event":"device_update","data":{"id":"work2-fan-1","status":"on","last_changed":"..."}}
{"event":"power_update","data":{"total_w":340,"by_room":{"drawing":95,"work1":0,"work2":245}}}
{"event":"alert","data":{"id":7,"type":"after_hours","room":"work2","message":"...","triggered_at":"..."}}
```

- Client reconnect with exponential backoff; on (re)connect the frontend does one REST snapshot fetch, then applies deltas. This makes "no page refresh" robust.

## 8. Alerts Engine

Background task evaluating rules every 30 s (config):

| Rule | Condition | Alert type |
|---|---|---|
| After-hours | Any device ON outside 09:00–17:00 local | `after_hours` (per room, aggregated: "Work Room 2: 2 fans + 3 lights still ON at 22:04") |
| Marathon room | All devices in a room ON continuously > 2 h (from `state_events`) | `marathon_room` |
| (Stretch) Power spike | Office total > 90% of max for > 15 min | `high_load` |

Alerts have a lifecycle (`triggered_at`, `resolved_at`), are stored in an `alerts` table, timestamped in local time on display, pushed over WebSocket, **and** enqueued for the Discord bot's proactive announcer (bonus points). De-duplication: an active alert of the same (type, room) is updated, not re-created.

## 10. MCP Layer (FastMCP) + LangGraph Agent

### 10.1 Why MCP here
The rubric wants the bot to give "real answers from actual simulated data." MCP makes the grounding explicit and demo-able: the agent literally cannot answer without calling a tool that reads the backend. It's also a great architecture talking point in the video.

### 10.2 FastMCP server (`mcp_server/`)
Runs as HTTP (streamable) on its own port. Tools are thin async wrappers over the backend REST API (httpx client) — **not** direct DB access, preserving one source of truth path:

| Tool | Args | Returns |
|---|---|---|
| `get_office_status()` | – | full device snapshot grouped by room |
| `get_room_status(room)` | `"drawing"\|"work1"\|"work2"` (with alias resolution: "work room 1" → `work1`) | room devices + power |
| `get_power_usage()` | – | `{total_w, by_room, today_kwh}` |
| `get_active_alerts()` | – | active alerts list |
| `get_device_history(device_id, limit)` | – | recent state events (nice for "when was fan 2 last turned on?") |

Each tool returns compact JSON plus a `generated_at` timestamp so the LLM can phrase recency honestly.

### 10.3 LangGraph agent (`agent/`)
- LLM: DeepSeek `deepseek-chat` via OpenAI-compatible endpoint (`base_url=https://api.deepseek.com`), temperature ~0.6 for personality, `max_tokens` capped for Discord's 2000-char limit.
- Tools loaded from the MCP server via `langchain-mcp-adapters` (`MultiServerMCPClient`).
- Graph: standard ReAct loop — `agent → tools → agent → END`, with a hard cap of 3 tool iterations.
- System prompt contract: *"You are the office's friendly electricity watchdog on Discord. ALWAYS call a tool before stating any number. Never invent device states. Be warm, brief, lightly funny; the boss hates robotic data dumps. Use Watts/kWh with sensible rounding. Current office hours 9–5 Asia/Dhaka."*
- **Determinism guard (lesson from QueueStorm):** numbers shown to the user come from tool JSON; the LLM only wraps them in prose. A post-check regex verifies the reply's wattage total matches the tool output within rounding; on mismatch, fall back to a templated response. LLM failure/timeout (8 s) → templated plain response so commands never hang.
- Caching: `cachetools` TTL cache (5 s) keyed by command+room so `!status` spam doesn't burn tokens.

## 11. Discord Bot (`bot/`)

### 11.1 Commands

| Command | Behavior |
|---|---|
| `!status` | Agent → `get_office_status` (+ `get_power_usage`) → friendly one-paragraph summary per the example format |
| `!room <name>` | Alias-resolves name, agent → `get_room_status`; unknown room → helpful correction listing valid rooms |
| `!usage` | Agent → `get_power_usage` → "Total power right now: 340W. Today's estimated usage: 2.1 kWh." with a human touch |
| `!alerts` | Active alerts, humanized |
| `!help` | Static embed listing commands |
| `@OfficePulse <free text>` | Full agent mode — natural-language questions routed through LangGraph (differentiator for judging) |

### 11.2 Proactive alerts (bonus)
The bot polls `GET /api/alerts?active=true&since=<cursor>` every 20 s (simple + robust; no cross-process WebSocket needed). New alert → agent generates a playful ping to `ALERT_CHANNEL_ID`: "⚠️ Hey! Work Room 2 still has 2 fans and 3 lights ON and it's 10 PM. Did someone forget to leave?" Resolved alerts get a follow-up ✅. Rate-limit: max 1 message per (type, room) per 30 min.

### 11.3 Resilience
Backend unreachable → apologetic canned reply, never a stack trace. All handlers async; long agent calls wrapped in `async with channel.typing()`.

## 14. Hardware Schematic Deliverable (Wokwi, one room)

Concept: an **ESP32** monitors (not controls) 5 AC devices in one room.

- **State sensing:** each device's live wire passes through a **ZMCT103C current transformer** (or ACS712 module in Wokwi's parts palette); rectified/conditioned output → ESP32 ADC pins (GPIO 32–36). Current above threshold ⇒ device ON; analog value also gives approximate power draw (fulfills the "optionally sensing current draw" line).
- **Wokwi practicality:** Wokwi lacks CT sensors, so the simulation stands in with 5 slide switches (device on/off) + 5 potentiometers on ADC pins (variable current), plus 5 LEDs mirroring detected state — and the README maps each simulated part to its real-world counterpart. This is exactly the "representative circuit that makes physical sense" the doc asks for.
- **Isolation note in README:** CTs/opto-isolators keep mains AC galvanically separated from the 3.3 V ESP32 — judges love seeing you thought about safety.
- ESP32 firmware sketch (Arduino): read ADCs every 2 s → JSON over Wi-Fi HTTP POST to `POST /ingest` (documented as the future real-hardware entry point that would replace the simulator — this ties the schematic into the system diagram beautifully).
- Deliver: public Wokwi project link + screenshot in `diagrams/`.
